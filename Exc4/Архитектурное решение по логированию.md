## Мотивация
* Улучшение наблюдаемости системы
* Сокращение время разбора ошибок и нештатных ситуаций
* Аудит и соблюдение регуляторных норм
* Мониторинг и alerting
* Анализ пользовательского опыта

Внедрение мониторинга требует ресурсов команды, поэтому начинать требуется с проблемного сервиса MES,
далее используемые им ресурсы - MES API, S3, Postgres, Rabbit.
Завершаем оставшимися компонентами Системы - CRM, CRM API, Shop и Shop API


## Предлагаемое решение
В компании нет устоявшейся решения для логирования.
Мне более знаком и привычен стек Grafana+Loki, использую его. <br/>
[Предлагаемая схема логирования](./exc4.drawio)

Сбор логов через Grafana Loki, поиск логов через Grafana и визуализация состояния Системы через Grafana Dashboards.
Сбор логов из файлов с использованием promtail и fluentbit, зависит от сценария применения
Настройка alerting на Grafana - по ошибкам и threshold на события Системы

Настройка ролевой модели. Доступ до сырых логов только для ограниченной группы пользователей (администраторов).
Доступ к dashboard на основании ролевой модели

Чувствительная информация должна маскироваться

Каждая из подсистем будет логировать в отдельную метку

Необходимые логи
Info:
* Изменение статуса заказа - timestamp, идентификатор заказа, предыдущий статус, текущий статус, время обработки
* Расчет стоимости заказа - timestamp, идентификатор заказа, время расчета, параметры расчета, этапы расчета, взаимодействие с компонентами (внутренними и внешними)
* Создание заказа через API - timestamp, идентификатор заказа, время расчета

Warning:
* Запросы по API, которые не были приняты из-за некорректности данных
* Неудачи вызовов API, постановку в очередь для которых определены политики повторов
* Бизнес ошибки, которые ожидаемы для Системы и обрабатываются им
* Долгие запросы на базе данных
* Предупреждения от базы данных, тк outdated index, old backup и т.п.

Error:
* Неуспешные ответы API 4XX, 5XX
* Необработанные ошибки Системы (unhandled exceptions)
* Проблемные сообщения в очереди: Dead-letters и *_error очереди
* Блокировки на базе данных

## Сравнение систем логирования
| Критерий           | ELK                                                                            | OpenSearch                                                   | Splunk                                                                 | Grafana stack                                                                |
|--------------------|--------------------------------------------------------------------------------|--------------------------------------------------------------|------------------------------------------------------------------------|------------------------------------------------------------------------------|
| Лицензия           | ElasticSearch SSLP                                                             | Apache 2.0                                                   | Проприетарная                                                          | Grafana: AGPLv3<br> Loki и Promtail: Apache 2.0                              |
| Производительность | Высокая. Оптимизирован для работы с большим объемом данных                     | Высокая. Аналогично ELK, но с улучшениями от сообщества      | Высокая. Ориентирована на Enterprise,но требует значительных ресурсов  | Высокая, Loki оптимизирован на работу с большим объемом данных               |
| Масштабируемость   | Высокая. Поддерживает горизонтальное масштабирование и распределенные кластера | Высокая. Поддерживает горизонтальное масштабирование         | Высокая. Ориентирована на Enterprise,но требует значительных ресурсов  | Высокая                                                                      |
| Интеграция         | Широкая, множество plugin и интеграций через logstash и Beats                  | Широкая, совместима с ELK                                    | Очень широкая, поддержка множества систем и форматов данных            | Хорошая интеграция                                                           |
| Интерфейс          | Kibana, мощный и гибкий интерфейс для визуализации и анализа данных            | OpenSearch Dashboard: аналогичен Kibana с улучшениями сообщества | Мощный и удобный, высокий порог входа                                  | Grafana: удобный интерфейс для визуализации, менее специализирован для логов | 
| Управление данными | Гибкое, поддержка индексов, шаблонов, ротации данных и удаоения данных         | Гибкое, аналогичное ELK                                      | Гибкое, требует настройки. Поддержка индексов, ротаций и сжатия данных | Упрошенное, использование меток для индексации                               |

Я выбрал Grafana, как легкое и экономичное решение, с которым у меня был опыт

Оно по части функционала уступает ELK и OpenSearch. 
Решение ELK имеет ограничения по лицензированию.

Splunk, с ним был опыт для поиска логов, но решение дорогое. Для этого проекта не применимо.
