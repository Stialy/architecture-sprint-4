## Текущее положение дел

* Система состоит из CRM (UI+API+Backend+DB), MES (UI+API+Backend+DB)
* MES продукт разработан внешним поставщиком, куплен вместе с кодовой базой
* Открыт API MES для приема B2B заказов
* Взаимодействие CRM и MES через шину RabbitMQ
* Нагрузка растет на 100 заказов в месяц При приеме заказа производится расчет, 
среднее время расчета - 2-3 минуты, для сложных заказов время расчета может достигать 30 минут
* Операторы отслеживают новые заказы, взятие заказа влияет на их доход
* Поступают жалобы, что заказ обрабатывается долго, теряются. Увеличение числа заказов, 
приводит к увеличению негативных отзывов

## Существующие и потенциальные проблемные места
* Проблемы есть и ситуация деградирует
* Клиенты стали уходить, в том числе и крупные клиенты
* Нет систем мониторинга/логирования/трейсинга
* MES долго формирует список новых заказов на UI
* Задержки релизов, проблемы с циклами устранения найденных критических ошибок в релизах
* Не понятны возможности горизонтального масштабирования Системы. В системе работает по одному инстансу каждого из приложений
* Нагрузочное тестирования не производится
* Тестирование, в основном, ручное. Внедрены автоматические unit test при PR в dev-окружение
* Развертывание в release и production окружение ручные
* В MES, поиск заказа для отображения (read) влияет (тормозит) операции загрузки (write). Фильтры на поиск и пагенация не решили проблему отображения
* UI MES написан на React, у команды нет достаточного опыта с ним
* По диаграмме MES API выглядят как синхронные, что при времени обработке заказа является существенной проблемой

## Инициативы, которые необходимы для устранения нежелательных ситуаций
Краткосрочные
Стабилизация Системы, если это возможно, определение проблемных мест, проработка целевого дизайна и плана трансформации

* Внедрение мониторинга и логирования
* Внедрение трейсинга
* Внедрения процедур анализа данных мониторинга, логов и трейсинга
* Анализ возможности масштабирования существующего решения. Если MES зависит только от базы (stateless), то можно рассмотреть горизонтальное масштабирование существующего решения, новые инстансов для балансировки нагрузки (расчета).
    Разделения потока получения данных и расчета заказа. На этом этапе будет достаточным балансировка нагрузки на load balancer (nginx подобный)
* Внедрение ночных сборок с прогоном unit-тестов, для раннего определения проблем. Устранения найденных проблем, без накопления технического долга.
Сокращение цикла релизов. Требует дополнительного анализа, что приводит к увеличению цикла. Одним из вариантов может быть сокращение времени с внесения изменений в кодовую базу и попаданием на тестирование
* Проведение нагрузочных тестов, для выявления проблемных компонент
* Выявление узких мест системы

В перспективе полугода

Принимая во внимание текущую ситуацию, размер команды считаю, что в перспективе полугода можно произвести стабилизацию текущего решения (см. выше)
и перейти к улучшениям наиболее проблемной части системы - MES.
По имеющимся данным, проблема не в Базе данных MES, предпосылок для разделения Базы данных на часть для чтения и часть на запись на этом этапе нет.

* Разделение сервиса MES на
    * Сервис расчета стоимости
    * Сервис поиска данных    
    * Горизонтальное масштабирование решения
    * AsyncAPI
    * Добавление кэширования на поиске новых заказов, со стратегией Write-Thought при размещении заказа и инвалидацией кэша при назначении оператора на заказ
    * Улучшение навыков работы с React, для будущих улучшений UI Системы

Остальная часть Системы в перспективе полугода сохраняется как есть
На Диаграмме отображена только изменяемая часть Системы

```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

title Целевая архитектура проекта "Александрит" через полгода, Компонент MES

AddRelTag("brocker_message", $lineStyle = DashedLine())
AddRelTag("3d_models", $lineColor = "Orange")

Person_Ext(api_user, "B2B клиент", "Засчитывает стоимость заказа, размещает его через внешний API")
Person(operator, "Оператор", "Берет заказы в работу")

System_Boundary(jewerly_store, "Подсистема MES") {
    System_Boundary(mes, "MES") {
        Container(mes_web, "MES. Web приложение.", "Typescript, React", "Предоставляет интерфейс для операторов и список заказов")
        Container(mes_api, "MES API", "Kong", "MES API")
        Container(mes_write, "MES Сервис приема заказа", "C#, .NET", "Прием заказа")
        Container(mes_cache, "MES Cache", "Redis", "Write through")
        Container(mes_read, "MES Сервис статусов заказов", "C#, .NET", "Статус заказа")
        ContainerDb(mes_db, "MES Database", "PostgreSQL", "Заказ, оператор, статус")
    }
    
    Container(3d_files_storage, "Система хранения 3D моделей", "java, SpringBoot", "Хранит 3D модели")

    System_Boundary(calculator, "MES Calculator") {
        Container(mes_calculator, "MES Сервис", "C#, .NET", "Только расчет стоимости")
    }
    
    SystemQueue(message_broker, "Брокер сообщений", "RabbitMQ")
}

Rel(operator, mes_web, "Использует", "UI")
Rel(mes_web, mes_api, "REST")
Rel(api_user, mes_api, "Async REST")
Rel(mes_api, mes_write, "Сохранение")
Rel(mes_api, mes_read, "Чтение")
Rel(mes_write, mes_cache, "Обновление/сохранение")
Rel(mes_read, mes_cache, "Чтение/Запись")

Rel(mes_write, message_broker, "Постановка заказа на расчет")

Rel(mes_calculator, message_broker, "Получить заказ для расчета")

Rel(mes_write, 3d_files_storage, "Сохранение 3D моделей")
Rel(mes_calculator, 3d_files_storage, "Получение 3D моделей")

Rel(mes_cache, mes_db, "Операции с БД")

@enduml
```
