##  Мотивация

По описанию, Система не является высоконагруженной, скорость изменения данных не высокая
* количество заказов за единицу времени
* время расчета стоимости заказа значительное (2-3 минуты, для больщих заказов до 30 минут)
* Время производства значительное

При этом наблюдаются проблемы получения данных из Системы
Кэширование поможет снизить объем негативного опыта

В первую очередь, кэширование поможет для MES компоненты
* Получение списка активных заказов для оператора
* Справочники
* Состояние заказов и расчет их стоимости

## Предлагаемое решение
Добавить кеш на запись и на чтение, используя комбинацию паттернов ReadTrough/WrightTrough. При этом кеш останется консистентным, а запросы на чтение будут выполняться быстрее.

При добавлении заказа используем подход WriteThought, заказ добавляется в кэш и сохраняется в БД. 
При окончании расчет заказа, обновляется кэш и расчет заказа сохраняется в БД.
При отборе заказа оператором, обновляем кэш и сохраняем изменение в БД.
Тем самым обеспечиваем консистентность данных.

Создание заказа

```PlantUML
@startuml

participant "MES API"
participant MES
participant S3
participant Cache
participant Database

note over "MES", Database : Запись производится синхронно, чтобы гарантировать консистентность

"MES API" -> MES : order <id>
activate MES

MES -> S3: Сохранение описания заказа
activate S3

S3 -> MES: Описание заказа сохранено
deactivate S3

MES -> Cache : create new order
activate Cache

Cache -> Database : create new order
activate Database

Database -> Cache : order created
deactivate Database

Cache -> MES : order created
deactivate Cache

MES -> "MES API" : order <id> 
deactivate MES
 
@enduml
```

Расчет и обновление кэша

```PlantUML
@startuml

queue RabbitMQ as Queue
participant MES
participant S3
participant Cache
participant Database

note over "MES", Database : Запись производится синхронно, чтобы гарантировать консистентность

Queue -> MES: Получение заказа на расчет
activate MES

MES -> S3: Получение описания заказа
activate S3

S3 -> MES: Описание заказа
deactivate S3

MES -> MES: Расчет стоимости заказа

MES -> Cache: Обновление заказа результатами расчета
activate Cache

Cache -> Database: Обновление заказа
activate Database

Database -> Cache: Заказ обновлен
deactivate Database

Cache -> MES: Заказ обновлен
deactivate Cache

MES -> Queue: Заказ рассчитан
deactivate MES


@enduml
```

Запросы на получения списка заказов и статусов производятся из кэша

```PlantUML
@startuml


actor "Оператор" as Operator
participant "MES API"
participant MES
participant Cache
participant Database

Operator -> "MES API": Запрос статуса или истории
activate "MES API"

"MES API" -> MES: Запрос данных
activate MES

MES -> Cache: Получение данных из кэша
activate Cache

Cache -> MES: Запрошенные данные в кэше
deactivate Cache

MES -> MES: Запрошенный диапазон больше чем период в кэше

group Если данные запрошены за больший диапазон, чем кэш
MES -> Database: Запрос данных
activate Database

Database -> MES: Запрошенные данные
deactivate Database

MES -> MES: Объединение данных из базы и кэша 
end


MES -> "MES API": Запрошенные данные
deactivate MES

"MES API" -> Operator: Запрошенные данные
deactivate "MES API"

@enduml
```

### Инвалидация кэша
Кэш обновлять по расписанию, при обновлении в кэш загружаются актуальные записи, которые в работе и исторические данные за период.

Расписание и глубина исторических данных конфигурируется
Для начала, используем обновление кэша раз в сутки, до начала рабочего дня. Основная нагрузка приходится на начало рабочего дня.
Глубина исторических данных - месяц + 5 дней, Срок производства + небольшая дельта на действия клиента с заказом.
